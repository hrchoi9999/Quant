# TASK 10. Redbot User Model and Internal Model Mapping
## (사용자용 모델 / 관리용 모델 매핑 문서화)

# A. TASK 문서용

## 1. 작업 목적
TASK 10의 목적은 지금까지 개발된 내부 관리용 모델 구조를
Redbot 서비스 관점의 사용자용 모델 체계로 번역하는 것입니다.

즉,
- 내부적으로는 `S2`, `S3`, `S4`, `S5`, `S6`, `Router`를 계속 운용하되,
- 사용자에게는 기술적 모델명이 아니라
  **이해 가능한 포트폴리오 추천 상품명**으로 제시할 수 있도록
  매핑 기준과 설명 구조를 고정하는 것이 이번 TASK의 핵심입니다.

이번 단계가 완료되면,
Redbot는 “퀀트 엔진 모음”이 아니라
**사용자가 선택 가능한 포트폴리오 추천 서비스**로 설명될 수 있습니다.

---

## 2. 작업 배경
현재까지 내부 모델 구조는 아래와 같이 정리되어 있습니다.

- 기존 주식 모델:
  - `S2`: 레짐 필터/게이트 + 펀더멘털(품질/성장) 기반, 방어/안정 성격
  - `S3`: 가격 피처 기반 가속/급등, 공격/상승장 탄력 성격
- 신규 국면별 모델:
  - `S4`: 상승 국면 전용
  - `S5`: 보합 국면 전용
  - `S6`: 하락 국면 전용 방어형
- 상위 통합 모델:
  - `Router`: 국면별로 어떤 모델을 쓸지 결정하는 상위 엔진

또한 장기 방향은 ETF를 포함한 멀티에셋 유니버스를 구축해
- 상승: 주식/섹터 베타 + 일부 공격 ETF
- 보합: 배당/저변동/커버드콜 + 채권/현금대체
- 하락: 채권/달러/금/인버스
로 포트폴리오를 구성하는 것입니다.

따라서 서비스 단계에서는,
이 내부 구조를 그대로 보여주지 말고
사용자가 이해 가능한 상품 체계로 정리해야 합니다.

---

## 3. 핵심 원칙
1. Redbot가 사용자에게 제공하는 것은 “엔진명”이 아니라 “포트폴리오 추천 상품”이다.
2. 내부 관리용 모델과 사용자용 모델은 반드시 구분한다.
3. 사용자용 모델 수는 적고 명확해야 한다.
4. 관리용 모델 수는 많아도 되며, 성과관리/운영통제를 위해 유지한다.
5. 사용자 설명은 투자성향, 기대 역할, 변동성 성격, 주 사용 자산 중심으로 작성한다.
6. 기술적 세부사항(ADX, RSI, Donchian, ATR 등)은 내부 설명에만 사용한다.
7. Router는 사용자용으로는 “자동전환형”으로 번역하는 것이 원칙이다.

---

## 4. 사용자용 모델 체계(권장안)
Redbot 사용자용 모델은 아래 4개를 표준안으로 정의합니다.

### 4.1 안정형
- 목표: 손실 방어 우선
- 설명: 채권/달러/금/현금대체 비중이 높은 보수형 포트폴리오
- 성격: 변동성 낮음, 수익보다 방어 우선
- 내부 연결: `S6` 중심 + 필요시 `Router(stable bias)`

### 4.2 균형형
- 목표: 수익과 안정의 균형
- 설명: 주식과 ETF를 함께 사용하되, 보합장/중립장에 적합한 균형형 포트폴리오
- 성격: 중간 변동성, 손실 방어와 수익 추구의 균형
- 내부 연결: `S2 + S5` 중심 + `Router(balanced bias)`

### 4.3 성장형
- 목표: 상승장에서 수익 확대
- 설명: 상승 추세를 적극 활용하는 공격형 포트폴리오
- 성격: 변동성 높음, 장기 수익 확대 지향
- 내부 연결: `S3 + S4` 중심 + `Router(growth bias)`

### 4.4 자동전환형
- 목표: 시장 상황에 따라 자동 대응
- 설명: 상승/보합/하락 국면에 따라 내부 모델을 자동 전환하는 동적 포트폴리오
- 성격: 규칙 기반 자동 운용
- 내부 연결: `Router(auto)`

## 5. 관리용 모델 체계
내부 운영에서는 아래 모델을 계속 관리합니다.

### 5.1 주식 슬리브
- `S2`: 안정형 주식 엔진
- `S3`: 공격형 주식 엔진

### 5.2 ETF / 멀티에셋 슬리브
- `S4`: 상승장용 ETF allocation
- `S5`: 보합장용 ETF allocation
- `S6`: 하락장용 ETF allocation

### 5.3 상위 엔진
- `Router`: 국면별 모델 선택/병합 엔진

### 5.4 참고 베이스라인
- `S3 core2`: 기존 비교용 베이스라인

# C. Codex 개발 기본방향

- 이번 TASK 10의 핵심은 기술 모델을 서비스 상품으로 번역하는 것이다.
- 사용자에게는 단순하고 명확한 4개 모델만 보여주고, 내부적으로는 S2/S3/S4/S5/S6/Router를 계속 관리한다.
- “내부는 복잡하게, 외부는 단순하게”가 기본 원칙이다.
